Expense Tracker — Kiến trúc microservices

Một app quản lý chi tiêu cá nhân + chia tiền nhóm, tách thành 6 microservice độc lập giao tiếp qua REST và message queue. Tài liệu này cho bạn bản đồ: service nào làm gì, file nào nằm ở đâu, và cách trả lời khi được hỏi.

6 microservices 4 Postgres (database-per-service) 1 RabbitMQ 1 Next.js frontend ≈ 2.4k dòng backend · 3.1k frontend NestJS · TypeScript · TypeORM · Docker Compose

02 Mỗi service làm gì & tại sao tách riêng

Đây là bảng bạn sẽ dùng nhiều nhất khi trình bày — mỗi service có một trách nhiệm rõ, và một lý do kiến trúc để tồn tại độc lập.

ServiceTrách nhiệmDBĐiểm "kể chuyện"
auth-service Đăng ký, đăng nhập, cấp & xác thực JWT. Tra user theo email cho service khác. auth Stateless auth — các service tự verify JWT bằng shared secret, không gọi lại auth.
transaction-service CRUD giao dịch, lọc/tìm/phân trang, summary, endpoint nhận CSV. transactions Event producer — phát transaction.created khi ghi.
import-worker Xử lý import CSV bất đồng bộ: 1 message/dòng, validate, gọi API tạo giao dịch. — Queue worker — chất liệu KEDA autoscale theo độ dài queue.
report-service Tổng hợp tháng: theo danh mục, theo ngày, so tháng trước. — CPU-heavy — chất liệu HPA autoscale theo CPU. Đọc qua API, giữ DB-per-service.
group-service Nhóm, thành viên, chi tiêu chung, thuật toán "ai nợ ai", settle up. groups Cross-service — gọi auth-service (internal key) để mời thành viên bằng email.
notification-service Lắng nghe event → lưu thông báo → REST cho chuông (unread, mark read). notifications Event consumer — decouple: producer không biết ai nhận.

04 File nào làm gì — 6 loại file NestJS

Khi leader hỏi "phần này code ở đâu?", đây là bản đồ tư duy. Mọi service đều theo đúng 6 vai trò này.

main.ts
Khởi động app. Bật CORS, đặt prefix /api, khởi động HTTP + (nếu là consumer) kết nối RabbitMQ, lắng nghe cổng.
*.module.ts
Lắp ráp. Khai báo controller/service/entity nào thuộc về nhau — NestJS dùng để "dependency injection". File gốc còn nối DB.
*.controller.ts
Cửa vào HTTP. Mỗi hàm = 1 route (@Post(), @Get(':id')…). Nhận request, kiểm tra quyền (guard), gọi service. Không chứa logic.
*.service.ts
Bộ não. Logic nghiệp vụ thật nằm đây — tính balance nhóm, tổng hợp report, phát event. Đây là file bạn nên đọc kỹ nhất để giới thiệu.
*.entity.ts
Bảng dữ liệu. Một class = một bảng Postgres. TypeORM tự tạo bảng từ đây.
dto/*.ts
Hợp đồng dữ liệu vào. Định nghĩa body request được phép có gì + luật validate (email hợp lệ, số > 0…). Sai là chặn ngay ở cửa.

06 3 luồng giao tiếp — hiểu 3 cái này là hiểu kiến trúc

Nếu chỉ nhớ 3 điều để nói khi phỏng vấn, hãy nhớ đây.

① Sync REST + Stateless JWT

auth-service cấp JWT. Các service khác tự verify token bằng cùng một secret — không cần gọi lại auth mỗi request. Đây là lý do hệ thống scale được: auth không thành nút thắt cổ chai.

② Async qua message queue (RabbitMQ)

Tạo giao dịch → phát event → notification-service nhận. Import CSV → mỗi dòng một message → worker xử lý nền. Producer không chờ, không biết ai nhận → service này sập không kéo sập service kia.

③ Database-per-service

Mỗi service sở hữu DB riêng, không service nào đọc thẳng DB của service khác. Cần dữ liệu chéo thì gọi API (report → transaction, group → auth). Giữ ranh giới rõ ràng, deploy độc lập.

07 Trả lời phỏng vấn — câu hỏi hay gặp

Câu trả lời ngắn gọn, đúng trọng tâm, dựa trên chính code này.

"Sao lại chọn microservices cho app nhỏ thế này?"

Thành thật: về nghiệp vụ thì monolith là đủ. Tôi chọn microservices để thực hành các pattern phân tán — DB-per-service, async messaging, service-to-service auth — làm nền cho việc deploy Kubernetes. Mỗi service có lý do tồn tại: worker để tách tải async, report để tách tải CPU.

"Các service nói chuyện với nhau thế nào?"

Hai kiểu: đồng bộ qua REST khi cần trả lời ngay (report gọi transaction, group gọi auth — dùng internal API key), và bất đồng bộ qua RabbitMQ khi không cần chờ (thông báo, import). Xem mục 06.

"Xác thực hoạt động ra sao?"

auth-service cấp JWT lúc login. Mỗi service có một jwt.strategy.ts verify token bằng shared secret (Passport). Với gọi máy-với-máy (worker→transaction, group→auth) tôi dùng một internal API key qua header thay vì JWT người dùng.

"Phần code nào bạn tâm đắc nhất?"

Thuật toán chia nợ nhóm trong group-service/groups.service.ts: tính net mỗi người (đã trả − phần phải chịu ± thanh toán), rồi greedy match chủ nợ ↔ con nợ để ra số giao dịch trả tối thiểu — giống Splitwise. Và luồng import CSV bất đồng bộ có retry.

"Nếu làm lại / cải thiện gì?"

Bỏ synchronize:true của TypeORM để dùng migrations (an toàn dữ liệu). Thêm test tự động. Và một API Gateway đứng trước thay vì cho browser gọi thẳng từng service — sạch hơn khi số service tăng.
